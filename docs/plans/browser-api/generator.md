# Generator

**Parent:** [browser-api.md](browser-api.md)

## Purpose

CLI tool that reads W3C/WHATWG spec files and emits typed C# source code. The generator is not shipped as a library — it's a development tool run on-demand.

## Input Sources

- `specs/idl/*.idl` — 337 WebIDL files from [w3c/webref](https://github.com/w3c/webref)
- `specs/css/*.json` — 124 CSS property data files from w3c/webref

## Pipeline

1. **Parse** WebIDL files → AST (interfaces, enums, dictionaries, typedefs, callbacks)
2. **Parse** CSS JSON files → property definitions (name, value grammar, initial, inherited)
3. **Resolve** cross-references (interface inheritance, partial interfaces, mixins)
4. **Transform** AST → C# model (apply naming conventions, type mapping, async detection)
5. **Emit** C# source files to `src/BrowserApi/Generated/`

## WebIDL Parser

Options:
- **Write a C# parser** from the [WebIDL grammar](https://webidl.spec.whatwg.org/#idl-grammar) — full control, no Node dependency
- **Use webidl2.js via Node** — proven parser, but adds a Node dependency to the tool
- **Use the pre-parsed JSON** from `specs/idlparsed/` in webref — already parsed, skip the parser entirely

Decision: TBD. The pre-parsed JSON approach is fastest to implement but adds coupling to webref's JSON schema.

## CSS Value Grammar Parser

CSS value syntax (e.g., `<length> | <percentage> | auto`) is its own grammar. Needs a parser to determine what C# type to generate for each property.

The CSS data JSON files include a `value` field with the grammar string. Options:
- Parse the grammar and map to C# types
- Start with a hand-curated mapping for common value types, expand later

## Emitter Design

Output one file per WebIDL interface/enum/dictionary. Use `partial class` everywhere so hand-written extensions survive regeneration.

Template for generated file:
```csharp
// <auto-generated/>
// Source: {spec-name}.idl
// Generated: {timestamp}

namespace BrowserApi.Dom;

public partial class Element : Node {
    // generated members
}
```
